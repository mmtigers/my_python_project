# システム全体設計書 (Integrated System Architecture)

## 1. システム全体概要

本システムは、家庭環境の自動化・ロギング、ゲーミフィケーションを通じたタスク管理、および外部データの自動収集とコンテンツ生成を統合管理する複合型プラットフォームです。全体は大きく3つの独立したサブシステムで構成され、互いに連携しながら自律的かつ非同期に稼働しています。

### 1.1. サブシステム構成

1. **MY_HOME_SYSTEM (バックエンドコア)**
   * **役割**: IoT機器の制御、環境データの収集・分析、各種API・Webhookの統合ルーティング。
   * **技術要素**: Python, FastAPI, Raspberry Pi (推測), 各種外部API (SwitchBot, Nature Remo, LINE等)。

2. **Family Quest (フロントエンドUI)**
   * **役割**: 家族のタスク（クエスト）をRPG風に管理し、モチベーションを向上させるクライアントアプリケーション。
   * **技術要素**: React, TypeScript, Vite。

3. **DDD (データ処理バッチ・コンテンツ生成)**
   * **役割**: 特定プロジェクト（Secret Wallet等）に向けたデータ自動収集、プロンプト生成、および動画・音声のバッチダウンロード処理。
   * **技術要素**: Python, yt-dlp, Discord API等。

## 2. サブシステムA: MY_HOME_SYSTEM (バックエンドコア)

すべてのハードウェア制御、データロギング、外部サービス連携を担うシステムの心臓部です。

### 2.1. コンポーネント一覧と役割

#### A. コアサーバー・ルーティング機構
* **`unified_server.py` / `system_router.py`**:
  各種APIリクエストの受け口となり、処理を各ドメインサービスへ委譲する統合エントリーポイント。
* **`webhook_router.py`**:
  外部システムからのPush通知（Webhook）を受け取り、適切なハンドラへルーティングする。
* **`bounty_router.py`**:
  報酬システムに関するAPIルーティング。

#### B. ハードウェア・IoT制御モジュール
* **`switchbot_service.py`**:
  SwitchBotデバイス（プラグ、ボット等）の制御およびステータス（電源状態、消費電力等）の取得。
* **`nature_remo_monitor.py`**:
  Nature Remoを介した家電制御と、温度・湿度等の環境センサーデータの監視。
* **`camera_monitor.py`**:
  ネットワークカメラの稼働監視およびスナップショット取得。
* **`tv_lock_monitor.py`**:
  TVの稼働時間を監視し、規定時間を超えた場合のロック制御を行う。
* **`connect_speaker.py`**:
  スマートスピーカー等への音声出力・通知連携。

#### C. 外部サービス・通知連携
* **`line_logic.py` / `line_handler.py`**:
  LINE Messaging APIを利用した双方向通信。ユーザーからのメッセージ解析、メニュー表示、システムからの通知送信を行う。
* **`google_photos_service.py`**:
  Google Photos APIと連携し、特定アルバムへの写真アップロードや取得を行う。
* **`weather_service.py`**:
  外部の天気APIから気象情報を取得し、システム内の判断材料（例：雨の日の特定タスク発火など）に利用する。

#### D. AI・分析エンジン
* **`ai_service.py`**:
  LLMを活用した推論・テキスト生成の共通インターフェース。
* **`log_analyzer.py`**:
  蓄積された各種ログ（センサー、タスク消化、システムログ）のパターンを分析する。
* **`weekly_analyze_report.py`**:
  週次で家庭内の状況（健全性、タスク消化率など）をAIで要約し、レポートとして出力（LINE等へ送信）する。

#### E. クエストバックエンド (Family Quest用API)
* **`quest_router.py` / `quest_service.py` / `game_logic.py`**:
  フロントエンド(Family Quest)からのリクエストを処理。ユーザーのレベル計算、経験値(コイン)の付与・消費、ボス討伐状況の更新、報酬インベントリの管理を計算し、データベース(DB)へ永続化する。

#### F. インフラ・監視タスク (フェイルソフト機構)
* **`server_watchdog.py`**:
  サーバープロセスの死活監視。異常検知時の再起動処理など。
* **`nas_monitor.py`**:
  NASの接続状態、ディスク容量の監視。枯渇前の事前アラート発報。
* **`memory_monitor.py`**:
  システムのメモリ使用量を監視し、OOM (Out of Memory) を未然に防ぐ。
* **`cron_reporter.py`**:
  定期実行ジョブ（Cron）の実行結果をまとめ、管理者にレポートする。

### 2.2. 主要なデータフロー
* **センサー監視フロー**:
  `nature_remo_monitor` / `switchbot` ➡ `sensor_service` ➡ `database` に時系列保存 ➡ `weekly_analyze_report` が集計 ➡ LINEへ送信。
* **WebHook通知フロー**:
  外部システムからのPush ➡ `webhook_router` ➡ 内容解析 ➡ `line_handler` を通じてユーザーへ通知。

## 3. サブシステムB: Family Quest (ゲーミフィケーションUI)

MY_HOME_SYSTEMのクエストAPIと連携し、タスクをRPG風に管理するReactベースのSPA（Single Page Application）です。

### 3.1. コンポーネント構成と役割

#### A. アプリケーション基盤
* **`App.tsx`**:
  ルートコンポーネント。全体の表示モード(`viewMode`)、アクティブなタブ(`activeTab`)、選択中のユーザー(`currentUserIdx`)をStateとして一元管理し、各画面・モーダルをマウントする。
* **`main.tsx`**:
  Reactツリーのレンダリングおよび、React Query (QueryClient) などのプロバイダ設定を行うエントリーポイント。
* **`@/types` (推測)**:
  `User`, `Quest`, `Item` などのTypeScript型定義を管理。

#### B. クエスト・タスク管理機能
* **`QuestList.tsx` / `ApprovalList.tsx`**:
  日常のタスク（クエスト）を一覧表示。通常・特別・無限クエストを分類してレンダリングする。
* **`useQuestStatus` (カスタムフック)**:
  クエストの状態（未完了、申請中、完了）を管理・判定するロジック。

#### C. ユーザー・ファミリー状態管理
* **`UserStatusCard.tsx`**:
  選択中ユーザーの現在のレベル、HP、経験値プログレスバーを表示する。
* **`FamilyMileageCard.tsx`**:
  家族全体で共有するポイント（マイレージ）の状態を可視化する。
* **`BossCard.tsx`**:
  現在出現しているボスモンスターの情報（HP、属性など）と、攻撃ボタン（クエスト完了連携）を表示する。
* **`WeeklyTrends.tsx`**:
  直近のクエスト消化傾向や活動履歴をグラフ等で表示する。

#### D. 報酬・ショップ機能
* **`EquipmentShop.tsx` / `ShopContainer.tsx`**:
  獲得したコインを消費して装備品を購入するUI。
* **`RewardList.tsx`**:
  設定されたご褒美（リアルな報酬等）の一覧と交換用UI。
* **`InventoryList.tsx`**:
  現在所持しているアイテムや装備品の確認画面。

#### E. 共通UI・演出エフェクト
* **`Modal.tsx` / `ConfirmModal.tsx` / `LevelUpModal.tsx`**:
  汎用的なモーダルダイアログ群。
* **`BattleEffect.tsx`**:
  クエスト完了（ボス攻撃時）の視覚的な演出（斬撃エフェクトなど）を描画する。
* **`useSound` (カスタムフック)**:
  承認音、レベルアップ音、攻撃音などのSE（サウンドエフェクト）再生ロジック。

### 3.2. データフロー・状態管理
* **クエスト完了フロー**:
  1. ユーザーが `QuestList` でクエスト完了をタップ。
  2. `App.tsx` の `executeConfirm()` が発火。
  3. `useGameData.completeQuest()` 経由でバックエンド(`quest_router.py`)へAPIリクエスト。
  4. (成功時) `bossEffect` ステートが更新され、`BattleEffect` が描画される。
  5. APIから最新のユーザー・ボス状態を再フェッチし、UIを更新。

## 4. サブシステムC: DDD (データ処理バッチ・コンテンツ生成)

情報収集および特定のコンテンツ展開（Secret Wallet等）を支える独立したバッチスクリプト群です。

### 4.1. コンポーネント一覧と役割
* **`batch_download_discord.py`**:
  Discordの特定チャンネルからメディアURLを抽出し、一括ダウンロードする。リトライ制御(`HTTPAdapter`)を実装。
* **`extract_youtube_urls.py`**:
  指定されたYouTubeチャンネルやプレイリストからURLリストを抽出する。
* **`bot_cookies.txt`**:
  認証が必要なサイトからのダウンロードに利用するセッションクッキー情報。
* **`split_prompts.py`**:
  長大なプロンプトやシナリオテキストを論理的なチャンクに分割する。
* **`split_results/` ディレクトリ群**:
  「保育士」「看護師」といったペルソナ別、またはシチュエーション別に生成・分割された大量のMarkdownデータ群。AI画像生成やコンテンツ出力のソースとして機能する。
* **`data/known_casts.json`**:
  コンテンツ生成の対象となるキャスト（登場人物）のメタデータ管理ファイル。

### 4.2. データ処理フロー
* **自動ダウンロードフロー**:
  1. 定期実行（Cron等）で起動。
  2. 対象リスト (`list/` 内のテキストファイル等) を読み込み。
  3. `nas_monitor` の情報やローカル容量をチェック。
  4. `yt-dlp` 等のライブラリを用いてNAS上の指定ディレクトリへ保存。
  5. 実行結果を `history.txt` に追記。

## 5. システム間連携とデータフロー (統合ビュー)

3つのサブシステムは、主に**データベース（DB）**、**NAS（ストレージ）**、および**REST API**を介して連携しています。

1. **日常のロギングとゲーミフィケーションサイクル**
   * `[Family Quest (UI)]` ➡ クエスト完了API ➡ `[MY_HOME_SYSTEM (Backend)]`
   * バックエンドがDBを更新（ユーザー経験値UP、ボスのHP減少）。
   * 同時に、`[IoTデバイス]` ➡ 環境データをDBへ随時保存。
   * 週末に `weekly_analyze_report.py` がこれらを総合して分析し、家族の活動サマリーをLINEへ通知。

2. **ストレージリソースの協調**
   * `[DDD (Batch)]` は大量のメディアデータをNASへダウンロード。
   * `[MY_HOME_SYSTEM]` の `nas_monitor.py` がNAS容量を監視し、DDDの活動によって容量が逼迫した場合はアラートを発報、または一時的にバッチを停止させる。

## 6. 保守・拡張に向けた指針 (推奨事項)

コード構成の解析に基づき、今後の保守性・安定性を高めるためのポイントを抽出しました。

1. **APIインターフェースの型同期 (TypeScript ↔ FastAPI)**
   * `Family Quest` 内で `any` キャストが見受けられる箇所があります。FastAPIのOpenAPI (Swagger) スキーマからTypeScriptの型定義を自動生成（`openapi-typescript` 等を利用）することで、バックエンドとフロントエンド間のデータ構造の不一致によるランタイムエラーを防ぐことができます。

2. **ブラウザ標準APIの脱却 (Family Quest)**
   * 一部のコンポーネント（装備変更など）で `confirm()` が使用されています。JavaScriptのスレッドブロックを防ぎ、UXを向上させるため、実装済みの `ConfirmModal` コンポーネントへの置き換えを推奨します。

3. **バッチ処理の冪等性（べきとうせい）確保 (DDD)**
   * ダウンロードバッチ（`batch_download_discord.py`等）において、途中でプロセスがクラッシュした場合に、「どこまでダウンロードが完了したか」を正確に再開できる仕組み（DBでのステータス管理、またはファイルのハッシュチェック等）を強化することで、深夜帯の無人稼働の信頼性がより向上します。
